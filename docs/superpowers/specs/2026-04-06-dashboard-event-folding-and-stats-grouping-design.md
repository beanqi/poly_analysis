# Dashboard Event Folding And Stats Grouping Design

## Goal

优化监控面板的可读性：

- 事件列表默认只展示最近 10 个事件，其余事件折叠，允许用户按需展开
- 策略统计从单张扁平表调整为按买入价分组展示
- 统计口径保留现有 `10/20`、`20/40`、`30/60`，并新增 `30/70`、`30/80`、`30/90`

## Current State

- `/api/stats` 当前通过 `DEFAULT_BUY_THRESHOLDS` 和 `DEFAULT_SELL_THRESHOLDS` 做笛卡尔积，再过滤 `sell > buy`
- 前端事件列表直接平铺渲染全部事件
- 前端策略统计直接渲染扁平 `rows` 表格，展示顺序为阈值组合 + 时间段 + Outcome

当前问题：

- 历史事件一多，事件列表信息密度过高，最近事件不够突出
- 统计表将“策略组合”和“时间段”混在同一维度里，阅读成本高
- 新需求只希望补充指定策略组合，而不是扩大为更多自动组合

## Design Decisions

### 1. 策略统计改为显式策略组合

后端不再依赖买入阈值与卖出阈值的笛卡尔积生成组合，而是定义固定的策略组合列表：

- `10/20`
- `20/40`
- `30/60`
- `30/70`
- `30/80`
- `30/90`

这样可以精确保留现有统计口径，并只新增用户要求的 3 组，不会意外出现 `10/40`、`10/60`、`20/60` 等未指定组合。

### 2. `/api/stats` 保留兼容行数据，并新增分组数据

`/api/stats` 继续返回现有 `rows` 字段，避免破坏现有消费者；同时新增 `groups` 字段，供前端直接渲染分组视图。

`groups` 结构按买入价分组，每个分组下包含多个卖出策略卡片；每个卡片下再按时间段列出 `Yes`、`No`、`Combined` 三类统计。

排序规则固定为：

- 分组按 `buy_threshold_cents` 升序
- 同组内策略按 `sell_threshold_cents` 升序
- 时间段按 `DEFAULT_BUCKETS` 顺序
- 每个时间段内的行顺序固定为 `Yes`、`No`、`Combined`

目标结构示意：

```json
{
  "meta": {
    "event_count": 12,
    "events_with_samples": 8
  },
  "rows": [],
  "groups": [
    {
      "buy_threshold_cents": 30,
      "strategies": [
        {
          "label": "30买 / 70卖",
          "sell_threshold_cents": 70,
          "buckets": [
            {
              "bucket": "0-1m",
              "rows": [
                { "outcome": "Yes" },
                { "outcome": "No" },
                { "outcome": "Combined" }
              ]
            }
          ]
        }
      ]
    }
  ]
}
```

### 3. 事件列表默认只展示最近 10 个事件

前端将事件列表拆分为：

- 默认可见事件：最近 10 个
- 折叠事件：其余历史事件

页面新增一个展开/收起按钮：

- 默认文案：`展开其余 N 个事件`
- 展开后文案：`收起历史事件`

如果当前选中的事件不在前 10 个中，前端应自动展开历史事件，避免“选中项不可见”。

最近事件的判定直接沿用 `/api/events` 的返回顺序，即 `start_ts` 倒序后的前 10 条。

### 4. 策略统计按买入价分组展示

前端将“策略统计”从一张总表改为分组块状布局：

- 第一层：按买入价分组，例如 `10买入`、`20买入`、`30买入`
- 第二层：每个买入价下面按卖出价展示策略卡片，例如 `30买 / 60卖`、`30买 / 70卖`
- 第三层：每张卡片内部保留表格，按时间段展示 `Yes / No / Combined`

这样可以让用户先定位买入价，再比较同一买入价下多个卖出阈值的差异，尤其适合 `30` 买入下新增的 `70/80/90` 三组。

## Data Flow

1. 后端使用固定策略组合列表执行事件回放统计
2. 后端先生成原始 `rows`
3. 后端基于 `rows` 生成 `groups`
4. 前端请求 `/api/stats` 后优先使用 `groups` 渲染
5. 前端请求 `/api/events` 后只展示前 10 个事件，其余事件默认折叠

## Error Handling

- 若 `groups` 为空，前端显示空状态，不渲染策略分组卡片
- 若事件总数不超过 10，前端不显示展开/收起按钮
- 若当前选中事件在刷新后不存在，则仍沿用现有逻辑回退到默认事件

## Testing

后端自动化测试：

- 更新 `tests/test_analytics.py`
  - 验证固定策略组合只包含 6 组目标组合
  - 验证 `30/70`、`30/80`、`30/90` 会出现在统计结果中
  - 验证 `groups` 会按买入价聚合并包含预期卖出策略
- 更新 `tests/test_api.py`
  - 验证 `/api/stats` 返回 `groups`
  - 验证 `30` 买入分组下包含 `60/70/80/90` 四个卖出策略

前端手动验证：

- 事件数大于 10 时，默认只展示 10 个事件
- 点击按钮后可以展开并再次收起历史事件
- 当选中的事件位于历史事件区时，列表自动保持展开
- 策略统计按买入价分组展示，且 `30` 买入下可见 `60/70/80/90` 四张策略卡片

## Scope

本次只调整展示结构和统计组合，不修改数据库结构，不新增筛选条件，不引入新的前端框架或构建流程。
